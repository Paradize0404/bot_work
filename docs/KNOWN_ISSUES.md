### ImportError: неверное имя при импорте из `db.engine` → краш бота (2026-02-28)
- **Симптом:** `ImportError: cannot import name 'async_session' from 'db.engine'. Did you mean: 'AsyncSession'?` — бот не запускается.
- **Причина:** В проекте сессия называется `async_session_factory`, а не `async_session`. Имя угадывалось по аналогии с другими проектами без проверки реального файла.
- **Решение:** Исправлена одна строка в `use_cases/pnl_sync.py`: `from db.engine import async_session_factory as async_session`.
- **Как избежать:**
  1. **PROJECT_MAP.md** теперь содержит раздел **🔌 Public API ключевых модулей** с точными именами.
  2. **`db/engine.py`** теперь содержит `__all__` — Python сам выдаёт явную ошибку при неверном имени.
  3. DoD-чеклист (PROJECT_MAP.md) содержит пункт: «Имена импортов сверены с таблицей Public API».
  4. Перед написанием нового модуля всегда делать `grep_search` по файлу-источнику или проверять `__all__`.

### CI падает на шаге `black` после правки кода (2026-02-23)
- **Симптом:** Коммит с логической правкой проходит все lint/test шаги, но падает на `Check formatting with black`. Тесты даже не запускаются (pytest skipped).
- **Причина:** `black` — абсолютный форматтер: любое отклонение от его стиля (лишний пробел, неправильный отступ, длинная строка) = провал CI. При ручном написании/редактировании кода форматирование легко сбивается.
- **Решение:** Перед каждым коммитом запускать `black` на изменённых файлах:
  ```bash
  python -m black use_cases/имя_файла.py
  # или на весь проект:
  python -m black .
  ```
  После этого `git add` + `git commit`.
- **Как избежать (AI-ассистент):** После любой правки `.py`-файлов — **обязательно** запустить `python -m black <файлы>` перед `git add`. Это часть рабочего процесса коммита, не опциональный шаг.
- **Как избежать (разработчик):** Настрой VSCode: `"editor.formatOnSave": true` + расширение `ms-python.black-formatter` → файл форматируется автоматически при сохранении.
- **Проверка без изменений файла:** `python -m black --check .` — выведет список файлов, требующих форматирования, не изменяя их.

### Авто-перемещение: пропускало товары с `main_unit = NULL` (2026-02-23)
- **Симптом:** В уведомлении `🚚 Авто-перемещение расх.мат.` строка вида `Клиническая: 2 перем. (пропущено 4 товаров)`. Товары с отрицательным остатком в iiko — есть, но в перемещение не попадают.
- **Причина 1:** `iiko_product.main_unit = NULL` — некоторые товары в iiko не имеют зафиксированной «базовой единицы» (поле `mainUnit` в REST API). Код `negative_transfer.py` ожидал UUID единицы измерения из БД и пропускал товар без неё.
- **Причина 2 (MAIN):** `iiko_product.name` хранится с **trailing spaces** (имена из REST API содержат пробел в конце), а OLAP возвращает те же имена БЕЗ пробел. `Product.name.in_(names)` — нет совпадений.
- **Причина 3 (вторичная):** Один и те же товары, пропущенные и на складе Бар и на Кухне, суммировались в `skipped_products` с дублями → счётчик завышен.
- **Решение:**
  - `_collect_negative_items` теперь сохраняет `measure_unit_name` из OLAP-поля `Product.MeasureUnit`.
  - Добавлена `_load_unit_id_by_name()` — fallback-запрос в `iiko_entity (root_type=MeasureUnit)` по имени единицы.
  - `_load_products_by_name` — **двухпроходный поиск**: проход  1 = точное `name.in_()`, проход 2 = `func.trim(name).in_()` для ненайденных (trailing spaces из iiko REST API).
  - При `main_unit = NULL` в БД — ищем UUID в `iiko_entity` по имени из OLAP. Если найден — используем, логируем INFO. Если нет — пропускаем с WARNING.
  - `skipped_products` дедуплицируется per-restaurant через `if pname not in list`.
- **Как избежать:**
  - Когда сравниваешь имена из OLAP с именами в БД — всегда делай `.strip()` на обеих сторонах.
  - iiko REST API сердить может пробелы в конце имени, OLAP — нет. Фаллбэк `func.trim()` обязателен в подобных поисках.
  - Когда для iiko-документа нужен UUID единицы измерения — всегда иметь fallback через `iiko_entity`. Не рассчитывать только на `product.main_unit`.

### Мутабельный default `default=[]` в ORM Column (2026-02-22)
- **Симптом:** Все InvoiceTemplate делят один и тот же список items — добавление товара в один шаблон появляется во всех.
- **Причина:** `Column(JSONB, default=[])` — Python вычисляет `[]` ОДИН раз при загрузке модуля. Все экземпляры получают ссылку на ОДИН объект.
- **Решение:** `default=list` (callable) — вызывается КАЖДЫЙ раз, создаёт новый список.
- **Как избежать:** НИКОГДА не писать `default=[]`, `default={}`, `default=set()` в Column/mapped_column. ВСЕГДА `default=list`, `default=dict`.
- **Проверка:** Grep `default=\[\]` по всем `.py` файлам.

### TOCTOU гонка в sync_lock (2026-02-22)
- **Симптом:** Редко, но два sync-процесса стартуют одновременно → дубли данных или deadlock.
- **Причина:** `if lock.locked(): return` + `async with lock:` — между проверкой и захватом другая корутина может захватить лок.
- **Решение:** `lock.acquire_nowait()` — атомарная проверка + захват в одной операции.
- **Как избежать:** НИКОГДА `if lock.locked()` + `async with lock`. ВСЕГДА `acquire_nowait()` или `async with lock` (если можно ждать).

### datetime.now() ≠ Калининградское время (2026-02-22)
- **Симптом:** Между 00:00 и 02:00 по Калининграду операции записываются с вчерашней датой.
- **Причина:** `datetime.now()` возвращает серверное время (UTC на Railway). В 01:00 KGD сервер считает что ещё вчера.
- **Решение:** `now_kgd()` из `use_cases._helpers` — всегда возвращает калининградское naive-время.
- **Как избежать:** НИКОГДА `datetime.now()`, `datetime.utcnow()` — ВСЕГДА `now_kgd()` или `_utcnow()`.

### Гонка данных в async-глобалах без Lock (2026-02-22)
- **Симптом:** Webhook handler иногда отправляет устаревшие/пустые остатки.
- **Причина:** Глобальные переменные (`_last_snapshot_hash` и др.) мутируются через `global` из конкурентных webhook-вызовов без синхронизации.
- **Решение:** `asyncio.Lock()` вокруг read-modify-write блоков.
- **Как избежать:** Если мутируешь глобальное состояние из async-кода — ВСЕГДА через `asyncio.Lock()`.

### REAL (float32) для денежных полей → потеря точности (2026-02-22)
- **Симптом:** Суммы в OCR-документах немного не совпадают с оригиналом (например, 12345.67 → 12345.6699219).
- **Причина:** `REAL` = IEEE 754 float32 = ~7 значащих цифр. Достаточно для координат, но НЕ для денег.
- **Решение:** `Numeric(12, 4)` — точное десятичное представление с 4 знаками после запятой.
- **Как избежать:** НИКОГДА `REAL`/`FLOAT` для денег/количеств. ВСЕГДА `Numeric(precision, scale)`.

### Кодировка файлов: UTF-16 → все русские символы = U+FFFD (2026-02-21)
- **Симптом:** В `bot/handlers.py` все русские буквы и emoji заменились на `�` (6444 символов). Синтаксис Python валиден, бот запускается, но все F.text-фильтры не срабатывают (кнопки меню не работают), все сообщения пользователю — мусор.
- **Причина:** Редактор (или автоформатировщик) пересохранил файл в кодировке UTF-16. При коммите git не выдал ошибку — он принял файл как бинарный. При чтении Python открыл его как UTF-8 и молча заменил каждый «сломанный» байт на `�`.
- **Почему долго чинилось:** Нет предыдущего чистого коммита для `git checkout`. CP1251-терминал падал с `UnicodeEncodeError` при любом `print()` с русскими символами (в том числе `→`), из-за чего скрипты восстановления крашились до сохранения файла. Каждое частичное сохранение смещало номера строк.
- **Решение:** Восстановление построчно по контексту кода + дампы repr() в промежуточные файлы. Суммарно ~10 итераций.
- **Как избежать:**
  1. `.gitattributes` — добавлен: `*.py text eol=lf encoding=utf-8`. Git теперь откажется коммитить Python-файл не в UTF-8.
  2. `.editorconfig` — добавлен: `charset = utf-8` для всех файлов. Поддерживающие редакторы (VSCode, PyCharm, Sublime) не смогут сохранить файл в другой кодировке.
  3. CI-шаг `Check UTF-8 encoding` — добавлен в `.github/workflows/ci.yml`. Сканирует все `.py`-файлы на наличие `�` и падает с явной ошибкой ещё до запуска тестов.
- **Правило:** Если видишь `�` в исходнике — не пытайся отлаживать логику. Сначала восстанови кодировку.

### iiko API: assemblyCharts — поля техкарт (2026-02-20)
- **Симптом:** Себестоимость блюд нереалистична — одни занижены в 5–10 раз, другие завышены
- **Причина 1:** Для `preparedCharts` использовалось поле `"amount"` — оно **не существует** в ответе API. Нет ошибки — просто `None`, умноженное на цену = 0.
- **Причина 2:** Суммарная стоимость ингредиентов не делилась на `assembledAmount`. Поле `assembledAmount` в теле техкарты = сколько единиц производит рецепт. Если `assembledAmount = 0.165` (165 г), то стоимость партии ингредиентов ≠ стоимость 1 кг.
- **Причина 3:** При фильтре по складам подразделения 89 товаров (п/ф, производственные) пропадали из СЦС — их нет на складах подразделения.
- **Решение:**
  - `amountOut` для обоих типов техкарт (показывает количество ингредиента после потерь при обработке)
  - `total_cost / assembledAmount` для получения стоимости единицы
  - Двухуровневый fallback: dept-склады → all-stores → накладные
- **Структура ответа assemblyCharts / preparedCharts:**
  - Уровень техкарты: `assembledProductId`, `assembledAmount`, `dateFrom`, `dateTo`, `items[]`
  - Уровень ингредиента: `productId`, `amountIn` (до обработки), `amountMiddle`, `amountOut` (после потерь), `amountIn1..3`, `amountOut1..3`
  - Поля `amount` и `amountOut` на уровне техкарты = `None` (не использовать!)
- **Как избежать:** На уровне техкарты использовать только `assembledAmount`. На уровне ингредиента — `amountOut`.

### FinTablo 429 Too Many Requests
- **Симптом:** Часть sync-задач падает с 429
- **Причина:** 13 параллельных задач без лимита → 500+ req/min, лимит FinTablo = 300 req/min
- **Решение:** `asyncio.Semaphore(4)` + retry с exponential backoff (2с → 4с → 8с → 16с → 32с)
- **Как избежать:** Любой внешний API с rate limit → семафор + retry с backoff с первого дня

### httpx: `Server disconnected without sending a response`
- **Симптом:** Случайные падения sync (особенно stores), ошибка `RemoteProtocolError`
- **Причина:** Транзиентные сетевые ошибки iiko сервера (TCP reset, timeout)
- **Решение:** `_get_with_retry()` — 3 попытки, backoff 1→3→7 сек

### OCR Pipeline: ложное предупреждение о несовпадении суммы при ставке НДС 22%
- **Симптом:** `⚠️ Позиция 3: сумма 850.0 не совпадает с расчётной 696.7` — хотя сумма верная
- **Причина:** `_VAT_RATE_MAP` не содержал `"22%"` → `_parse_vat("22%") = None` → `vat_dec = 0` → система считала expected как qty × price без НДС (696.7) и сравнивала с суммой с НДС (850.0)
- **Решение:** Добавить `"22%": Decimal("0.22")` в карту. Дополнительно: если ставка не в карте (`vat_unknown=True`) — не генерировать предупреждение о несовпадении, принимать сумму из документа как достоверную
- **Как избежать:** При добавлении новых ставок НДС (5%, 7% и т.д.) сразу добавлять их в `_VAT_RATE_MAP` в `use_cases/ocr_pipeline.py`

### OCR Pipeline: вторая страница многостраничной накладной не объединяется с первой
- **Симптом:** Двухстраничная накладная появляется как два отдельных документа в превью
- **Причина:** GPT на «Листе 2» не видит шапки с номером документа → `doc_number = null` → `group_key` не строится (`{inn}_{doc_number}_{date}` с null) → страница уходит в уникальный ключ `single_*`
- **Решение:** Двухпроходная группировка в `process_photo_batch()`. Проход 1: страницы с явным `group_key` + страница 1. Проход 2: «осиротевшие» страницы 2+ сопоставляются с существующими группами по `supplier_inn + date`. Этот подход работает т.к. ИНН поставщика и дата остаются одинаковыми на всех листах одного УПД
- **Как избежать:** Многостраничные документы — всегда загружать всё одновременно (альбомом). При разработке нового функционала группировки помнить: `group_key` может отсутствовать на страницах 2+, это нормально
- **Как избежать:** Ошибки Google Sheets API → `logger.warning`, никогда не `logger.debug`

### iiko API: 409 Conflict при отправке списания — "Cannot find WriteoffDocument by id" (2026-02-23)
- **Симптом:** `httpx.HTTPStatusError: Client error '409 Conflict'` + body `Cannot find WriteoffDocument document by id <UUID>`. Документ не отправлен.
- **Причина:** iiko возвращает 409 когда документ с данным UUID был ранее частично зарегистрирован (например, таймаут при прошлой попытке) и завис в нерабочем состоянии. Повторная отправка с тем же UUID всегда даст 409.
- **Решение:** В `adapters/iiko_api.py → send_writeoff()` при 409 генерируем новый UUID (`uuid4()`), обновляем `document["id"]` и `doc_id`, делаем retry. Одна повторная попытка с новым UUID позволяет пробиться через застрявшее состояние.
- **Как избежать:** При POST в iiko — если 409 + "Cannot find by id" = не логическая ошибка, а физическая проблема UUID. Решение всегда — новый UUID.

### get_user_requests: datetime не сериализуется в Redis FSM (2026-02-23)
- **Симптом:** `Object of type datetime is not JSON serializable` в `view_request_history` при `state.update_data(_history_cache=requests)`.
- **Причина:** `get_user_requests()` возвращала `created_at` как сырой Python `datetime`. Redis FSM (aiogram) сериализует состояние через `json.dumps()` — `datetime` не поддерживается.
- **Решение:** В `get_user_requests()` конвертировать `r.created_at.isoformat() if r.created_at else None`. В `format_request_text()` добавить ветку `isinstance(created, str)` → `datetime.fromisoformat(created).strftime(...)` для обратной совместимости с другими путями, где `created_at` остаётся `datetime`.
- **Как избежать:** Любой dict, который кладётся в `state.update_data()` — должен содержать только JSON-сериализуемые типы: `str`, `int`, `float`, `bool`, `list`, `dict`, `None`. `datetime`, `UUID`, `Decimal` — **запрещены** без конвертации.

### Аудит двойного нажатия: invoice + request handlers не были защищены (2026-02-23)
- **Симптом:** Быстрое двойное нажатие «✅ Отправить накладную» (invoice) или «✅ Отправить заявку» (request / dup) создавало две одинаковых операции. Отклонение заявки двумя администраторами одновременно → два уведомления создателю.
- **Причина:** `_sending_lock` был добавлен только в `writeoff_handlers.py`. `invoice_handlers.py::confirm_send`, `request_handlers.py::confirm_send_request` и `dup_confirm_send` добавляли `await callback.answer()` без блокировки — FSM-состояние сбрасывается ПОСЛЕ отправки, поэтому второй клик до завершения первого проходил фильтр и повторял операцию.
- **Решение:**
  - `invoice_handlers.py` — добавлен `_sending_lock: set[int]` + защита в `confirm_send` → делегирование в `_do_confirm_send`.
  - `request_handlers.py` — добавлен `_sending_lock: set[int]` + защита в `confirm_send_request` и `dup_confirm_send` → делегирование в `_do_confirm_send_request` / `_do_dup_confirm_send`.
  - `reject_request` — добавлен `_try_lock_request` + `finally: _unlock_request` (раньше было только chekc lock owner, но не захват лока).
- **Как избежать:** Каждый handler, который выполняет `create` / `send` операцию → обязательно `_sending_lock` (3 строки: check → add → finally discard). Паттерн из `writeoff_handlers.py` — эталон.
- **Итоговая карта защиты:**
  - `writeoff_handlers.py` — 3 send-handler → `_sending_lock` ✅; pending approve/reject/edit → DB `is_locked` ✅
  - `invoice_handlers.py` — `confirm_send` → `_sending_lock` ✅

### writeoff_handlers: double-click не блокировался
- **Симптом:** При быстром двойном нажатии «Отправить» документ создавался дважды
- **Причина:** `_sending_lock` проверялся (`if user_id in _sending_lock`), но никогда не устанавливался (`_sending_lock.add(user_id)` отсутствовал)
- **Решение:** Добавлен `_sending_lock.add(user_id)` + `finally: _sending_lock.discard(user_id)`
- **Как избежать:** Lock = 3 строки: check → set → finally discard. Если видишь только check — баг.

### IIKO_CLOUD_ORG_ID не задан → стоп-лист пуст
- **Симптом:** Кнопка «📛 Стоп-лист» показывает «Стоп-лист пуст», хотя в iiko позиции есть
- **Причина:** Старый standalone-скрипт (`main-stoplist.py`) использовал env `ORG_ID`, а бот ожидал `IIKO_CLOUD_ORG_ID`. На Railway была установлена только `ORG_ID`
- **Решение:** Добавлен fallback в `config.py`: `os.getenv("IIKO_CLOUD_ORG_ID") or os.getenv("ORG_ID")`
- **Как избежать:** При добавлении нового env — проверять совместимость имён с уже существующими переменными на Railway. Если данные те же — поддерживать оба имени через fallback
